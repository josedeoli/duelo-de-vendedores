Guia Completo

Uma ferramenta de programação agêntica que vive no seu terminal, entende o seu código e ajuda você a desenvolver mais rápido por meio de linguagem natural.

Visão geral

O que é Claude Code?

Claude Code é um agente de programação que opera diretamente no seu terminal. Diferente de um chatbot que apenas responde perguntas, o Claude Code lê arquivos, executa comandos, aplica mudanças e trabalha de forma autônoma através de problemas complexos — enquanto você observa, redireciona ou se afasta.

🗂️

Gestão de arquivos

Lê diretórios inteiros, cria e edita arquivos em todo o codebase, realiza refatorações em múltiplos arquivos.

🧪

Testes automatizados

Quando testes falham, Claude lê os erros, corrige o código e roda a suíte novamente até tudo passar.

🔀

Git e Pull Requests

Faz commits com mensagens descritivas, abre PRs e monitora pipelines de CI no GitHub e GitLab.

🛡️

Controles de segurança

O padrão é cauteloso: solicita permissão antes de modificar arquivos ou executar comandos.

🔌

Integrações MCP

Conecta-se a Figma, Notion, bancos de dados e outras ferramentas via servidores MCP.

⚡

Modo paralelo

Execute múltiplas sessões em paralelo para acelerar o desenvolvimento ou isolar experimentos.

Setup

Instalação

Claude Code requer Node.js 18+ e uma assinatura Claude Pro, Max ou acesso via API Anthropic.

Terminal 
$ npm install -g @anthropic-ai/claude-code

# Após instalar, faça login na sua conta Anthropic:
$ claude /login

# Verifique a instalação:
$ claude --version

# Inicialize o Claude no seu projeto:
$ claude /init
💡
Dica

Use /doctor a qualquer momento para diagnosticar e verificar sua instalação do Claude Code. Pressione f para que o Claude corrija os problemas encontrados automaticamente.

Workflow

Primeiros Passos

O fluxo de trabalho recomendado tem quatro fases. Separar exploração de implementação evita resolver o problema errado.

1

Explorar

Entre no modo de planejamento com /plan. Claude lê arquivos e responde perguntas sem fazer alterações.

Modo Plan 
> leia /src/auth e entenda como gerenciamos sessões e login.
> também veja como lidamos com variáveis de ambiente para secrets.
2

Planejar

Peça ao Claude que crie um plano detalhado de implementação. Pressione Ctrl+G para abrir o plano no editor e editá-lo antes que o Claude prossiga.

Modo Plan 
> quero adicionar autenticação Google OAuth. quais arquivos
  precisam mudar? qual é o fluxo de sessão? crie um plano.
3

Implementar

Saia do modo de planeajamento e deixe o Claude codificar, verificando em relação ao seu plano.

Modo padrão 
> implemente o fluxo OAuth do seu plano. escreva testes para o
  callback handler, rode a suíte de testes e corrija falhas.
4

Commit e PR

Peça ao Claude que faça commit com uma mensagem descritiva e crie um Pull Request.

Modo padrão 
> faça commit com uma mensagem descritiva e abra um PR

Segurança

Modos de Permissão

Por padrão, o Claude Code solicita permissão para ações que possam modificar seu sistema. Há três formas de reduzir essas interrupções mantendo o controle:

🤖

Modo Auto

Um modelo classificador revisa comandos e bloqueia apenas o que parece arriscado: escalada de escopo, infraestrutura desconhecida ou ações induzidas por conteúdo hostil.

✅

Allowlists

Permita ferramentas específicas que você sabe serem seguras, como npm run lint ou git commit. Configure via /permissions.

📦

Sandbox

Isolamento em nível de SO que restringe acesso ao filesystem e rede, permitindo ao Claude trabalhar com mais liberdade dentro de fronteiras definidas.

Exemplo de uso com modo auto 
$ claude --permission-mode auto -p "corrija todos os erros de lint"
Configuração

Arquivo CLAUDE.md

CLAUDE.md é um arquivo especial que o Claude lê no início de cada conversa. Inclua comandos Bash, estilo de código e regras de workflow. Isso dá ao Claude contexto persistente que não consegue inferir apenas do código.

ℹ️
Localizações do CLAUDE.md

~/.claude/CLAUDE.md — aplica-se a todas as sessões · ./CLAUDE.md — compartilhado com o time via git · ./CLAUDE.local.md — notas pessoais (adicione ao .gitignore)

Arquivo CLAUDE.md 
# Estilo de código
- Use ES modules (import/export), não CommonJS (require)
- Desestruture imports quando possível
- Prefer funções arrow para callbacks

# Workflow
- Faça typecheck ao terminar uma série de mudanças
- Prefira rodar testes individuais, não a suíte inteira
- Branches: feature/nome-da-feature, fix/descricao-do-bug

# Ambiente
- Node.js 20+, pnpm como gerenciador de pacotes
- .env.local para variáveis de ambiente em desenvolvimento

# Decisões arquiteturais
- Usar Prisma para ORM, não raw SQL
- API REST para endpoints públicos, tRPC para internos

# Importar arquivos adicionais:
Veja @README.md para visão geral e @package.json para comandos npm.

O que incluir vs. excluir

Incluir no CLAUDE.md
  • Comandos Bash que o Claude não consegue adivinhar
  • Regras de estilo que diferem dos padrões
  • Instruções de teste e runners preferidos
  • Etiqueta do repositório (nomes de branch, convenções de PR)
  • Decisões arquiteturais específicas do projeto
  • Variáveis de ambiente obrigatórias
  • Gotchas e comportamentos não óbvios
Não incluir no CLAUDE.md
  • Coisas que o Claude já descobre lendo o código
  • Convenções padrão que o Claude já conhece
  • Documentação detalhada de API (coloque um link)
  • Informações que mudam com frequência
  • Explicações longas ou tutoriais
  • Descrições arquivo-por-arquivo do codebase
  • Práticas óbvias como "escreva código limpo"
⚠️
Atenção

CLAUDE.md carregado toda sessão — mantenha-o enxuto. Se o Claude continua fazendo algo errado apesar de uma regra, o arquivo provavelmente está longo demais e a regra está se perdendo no ruído. Poda impiedosamente.

Performance

Gestão de Contexto

A janela de contexto do Claude contém toda a sua conversa — cada mensagem, arquivo lido e saída de comando. A performance degrada conforme a janela enche. A gestão de contexto é o recurso mais importante a ser administrado.

01

Use /clear entre tarefas não relacionadas

Sessões longas com contexto irrelevante reduzem a performance. Execute /clear para reiniciar o contexto completamente entre tarefas distintas.

02

Use /compact para sumarizar

Execute /compact Focus on the API changes para condensar o histórico mantendo as informações mais relevantes. Preserve o contexto de conversas longas sem resetar tudo.

03

Delegue investigações para subagentes

Quando Claude pesquisa um codebase, lê muitos arquivos — todos consumindo seu contexto. Use "use subagents to investigate X". Eles exploram em janelas de contexto separadas.

04

Escopo de investigações limitado

Nunca peça ao Claude para "investigar" sem delimitar o escopo. Isso pode levar ao Claude ler centenas de arquivos, enchendo o contexto. Seja específico: "leia apenas /src/auth".

05

Monitore com /context

Use /context para visualizar o uso atual da janela de contexto como uma grade colorida, com sugestões de otimização e avisos de capacidade.

Expertise

Melhores Práticas

1. Dê ao Claude formas de verificar o próprio trabalho

O Claude performa dramaticamente melhor quando pode verificar seu próprio trabalho — rodar testes, comparar screenshots, validar outputs. Sem critérios de sucesso claros, você se torna o único loop de feedback.

Antes vs. Depois dos prompts
EstratégiaAntes (vago)Depois (específico)
Critérios de verificação "implementa uma função de validação de e-mail" "escreva validateEmail. casos de teste: user@example.com é true, invalid é false. rode os testes após implementar"
Verificar UI "faça o dashboard parecer melhor" "[cole screenshot] implemente este design. tire screenshot e compare com o original. liste diferenças e corrija"
Bug fix "o build está falhando" "o build falha com este erro: [cole o erro]. corrija e verifique que o build passa. trate a causa raiz, não suprima o erro"

2. Forneça contexto específico nos prompts

Claude pode inferir intenção, mas não pode ler sua mente. Referencie arquivos específicos, mencione restrições e aponte para padrões de exemplo.

Prompts eficazes
  • Referencie arquivos com @ (Claude lê antes de responder)
  • Aponte para padrões existentes: "veja como HotDogWidget.php está implementado"
  • Descreva o sintoma, a localização provável e o que "corrigido" parece
  • Especifique o arquivo, cenário e preferências de teste
Prompts ineficazes
  • "adicione um widget de calendário" (sem contexto)
  • "corrija o bug de login" (vago demais)
  • "por que isso não funciona?" (sem logs ou código)
  • "melhore o código" (sem critérios definidos)

3. Corrija cedo e com frequência

Os melhores resultados vêm de loops de feedback curtos. Use Esc para parar o Claude imediatamente quando notar que está indo na direção errada. Contexto é preservado, então você pode redirecionar.

⚠️
Regra das duas correções

Se você corrigiu o Claude mais de duas vezes pelo mesmo problema em uma sessão, o contexto está poluído com abordagens fracassadas. Execute /clear e comece com um prompt mais específico que incorpore o que você aprendeu.

4. Deixe o Claude entrevistar você

Para features maiores, deixe o Claude entrevistar você primeiro. Isso revela coisas que você pode não ter considerado ainda — implementação técnica, UI/UX, edge cases e tradeoffs.

Prompt de entrevista 
> quero construir [breve descrição]. me entreviste em detalhes
  usando a ferramenta AskUserQuestion.

  Pergunte sobre implementação técnica, UI/UX, edge cases,
  preocupações e tradeoffs. Não faça perguntas óbvias.

  Continue entrevistando até cobrir tudo, depois escreva
  uma spec completa em SPEC.md.
Escala

Automação e Escala

Uma vez que você é eficaz com um Claude, multiplique seu output com sessões paralelas, modo não-interativo e padrões fan-out.

Modo não-interativo

Use claude -p "prompt" em pipelines CI, hooks de pre-commit ou qualquer workflow automatizado.

Exemplos de automação 
# Consulta única
$ claude -p "Explique o que este projeto faz"

# Saída estruturada para scripts
$ claude -p "Liste todos os endpoints de API" --output-format json

# Streaming para processamento em tempo real
$ claude -p "Analise este arquivo de log" --output-format stream-json

# Fan-out: migração em larga escala
$ for file in $(cat files.txt); do
    claude -p "Migre $file de React para Vue. Retorne OK ou FAIL." \
      --allowedTools "Edit,Bash(git commit *)"
  done

Padrão Writer / Reviewer

Use sessões paralelas para separar implementação e revisão — um contexto limpo melhora a revisão pois o Claude não está enviesado em relação ao código que acabou de escrever.

Sessão A (Writer)Sessão B (Reviewer)
Implemente um rate limiter para os endpoints da API —
— Revise a implementação do rate limiter em @src/middleware/rateLimiter.ts. Procure edge cases, race conditions e consistência com middlewares existentes.
Aqui está o feedback da revisão: [output da Sessão B]. Corrija esses problemas. —
Diagnóstico

Erros Comuns

!

A sessão "cozinha tudo junto"

Você começa com uma tarefa, pergunta algo não relacionado, volta à primeira tarefa. O contexto está cheio de informação irrelevante. Correção: use /clear entre tarefas não relacionadas.

!

Correções em loop

O Claude erra, você corrige, ainda está errado, corrige novamente. O contexto está poluído com abordagens fracassadas. Correção: após duas correções falhas, /clear e escreva um prompt melhor.

!

CLAUDE.md superespecificado

Se seu CLAUDE.md está muito longo, o Claude ignora metade porque as regras importantes se perdem no ruído. Correção: poda impiedosamente. Se já funciona sem a instrução, delete.

!

Confia sem verificar

O Claude produz uma implementação plausível que não trata edge cases. Correção: sempre forneça verificação (testes, scripts, screenshots). Se não pode verificar, não suba para produção.

!

Exploração infinita

Você pede ao Claude para "investigar" algo sem limitar o escopo. O Claude lê centenas de arquivos, enchendo o contexto. Correção: delimite investigações ou use subagentes para que a exploração não consuma o contexto principal.
